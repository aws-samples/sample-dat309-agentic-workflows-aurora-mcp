Wayfare · 2026 vision

A trip you talk into existence.

Wayfare is an agentic travel concierge: the user describes the trip in their own words — "five quiet days in Lisbon in May, walkable hotel, no red-eyes, decent vegetarian dinners" — and a supervisor of specialist agents searches, plans, books, and watches for disruptions. Every step is grounded in Aurora PostgreSQL, observable as a trace, and bounded by an explicit budget the user controls.

Pivot note
We dropped the product-catalog framing (overlap with another workshop) and the Apache AGE reference (Aurora PostgreSQL doesn't ship AGE). The "graph-feeling" queries — itineraries, connections, location hierarchies — are now expressed with Aurora-native primitives: pgvector for semantic, pg_trgm + tsvector for lexical, recursive CTEs for traversal, and logical replication + the Data API for streaming. One engine, several retrieval styles, no extension drama.
Chapter 01

Six pillars for Wayfare's narrative.

Each pillar maps a 2026-era agentic pattern onto travel — where the workflow is naturally long-running, multi-step, and high-stakes. That makes every pattern more visible than it would be in a search-and-buy demo.

Pillar 01

Durable memory of the traveler.

Aurora holds a traveler.facts table indexed by pgvector for semantic recall — passport country, loyalty status, dietary needs, prior trips, "I never sit in the back of the plane", aisle vs. window, things they've ruled out. Read on every turn, written by an explicit memory agent the user can audit.

Why it matters. Travel personalization compounds across trips. The first trip is fine; the fifth should be effortless because the agent already knows you.
Pillar 02

From a single chat to a supervisor.

One supervisor agent fans out to specialists: Flights, Lodging, Dining, Timing (weather/season), Calendar, Booking, Operations (disruption watch). Each owns a narrow MCP toolset and is independently versioned and replaceable.

Why it matters. A trip is many concerns held together. Specialist agents make that visible — and let teams own slices independently.
Pillar 03

MCP as the contract with the world.

Every external system — a GDS, OpenSky for live flight status, OpenWeather, a hotel API, Google Calendar, Stripe, the user's own Aurora — sits behind an MCP server with a versioned schema. A Tool Catalog page shows version, owner, p50 latency, last health check, and a dry-run button.

Why it matters. Travel is integrations all the way down. MCP turns "we wrote glue code" into "we shipped a contract."
Pillar 04

Governed autonomy, with money on the line.

Agents act under explicit budgets — total dollars, per-leg caps, refundable-only flags, calendar conflicts as hard stops. Anything that costs money pauses for an inline plan-and-confirm with a diff. The receipt links back to the trace that produced it.

Why it matters. "The agent booked the wrong flight" is the headline that kills the category. Show the leash; show that it works.
Pillar 05

Observability as a surface, not a sidebar.

The current Activity Panel becomes a Gantt-style trace: agents on the y-axis, time on the x-axis, tool calls as bars, with retries, cache hits, and confirmation gates inline. Every trace is permalinked and replayable in a sandbox.

Why it matters. Travel ops teams will use this view for debugging. Travelers will see a friendly view of the same trace as their itinerary timeline.
Pillar 06

Long-running workflows on Aurora.

A trip lives for weeks. Aurora persists every step of the plan as a state machine, the agent wakes on triggers (price drop, weather alert, gate change), and the user is told only the things that need their attention. Logical replication streams Aurora changes to the UI in real time.

Why it matters. 2026's agentic frontier is not single-turn — it's durable agents that watch the world for you. Aurora is a natural backbone for that.

Surface 07 · Long-running

Wakes the agent on disruptions.

The Operations agent watches your trip for weeks and acts only when it must. A gate change, a price drop, a weather flag — it tries to resolve the issue silently within scope, and asks only when it can't.
